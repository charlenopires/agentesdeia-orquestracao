O que é Inteligência Artificial?

A capacidade de um sistema de computador ou máquina de realizar tarefas que normalmente requerem inteligência humana, como percepção visual, reconhecimento de fala, tomada de decisão e tradução entre idiomas. Essa definição abrange uma ampla variedade de tecnologias e aplicações, incluindo aprendizado de máquina, redes neurais, processamento de linguagem natural e robótica.

Agentes de IA no ecossistema

Como funciona a IA Tradicional

A IA tradicional executa uma chamada única ao modelo de linguagem — single-call, reativa, sem estado.

Sem estado · Sem ferramentas · Sem decisão sobre o próximo passo

Como funciona o Agente de IA

O agente opera em um loop autônomo com ferramentas, memória persistente e decide o próximo passo a cada iteração.

Estado persistente · Ferramentas · Decide o próximo passo

O que é um agente?

Entidade computacional que percebe seu ambiente, raciocina sobre essas percepções, e age sobre o ambiente para alcançar objetivos.

  • Autonomia · opera sem intervenção humana direta
  • Reatividade · responde a mudanças no ambiente
  • Proatividade · toma iniciativa para alcançar objetivos
  • Adaptabilidade · aprende com experiências passadas

Definição LLM-Era

"Um sistema que usa um LLM para decidir o fluxo de controle de uma aplicação."

— LangChain

A diferença com sistemas tradicionais é precisa: numa cadeia RAG, os passos são fixos. Num agente, o LLM decide dinamicamente o que fazer a seguir.

O que torna algo um agente?

A definição mais clara vem da Anthropic: workflows seguem caminhos predefinidos de código; agentes direcionam dinamicamente seus próprios processos.

01

Três coisas diferentes

CHAMADA LLM

Single call

Um prompt → uma resposta. Sem loop, sem ferramentas, sem decisão sobre próximo passo.

prompt → resposta
WORKFLOW

Pipeline fixo

Múltiplas chamadas LLM, mas com fluxo predefinido em código. Determinístico, auditável.

a → b → c → d
AGENTE

Loop dinâmico

O LLM decide quais ferramentas chamar, quando parar, como adaptar. Probabilístico, flexível.

loop { decidir · agir · observar }

"Prefira workflows quando puder, agentes apenas quando precisar." — Anthropic

Anatomia do agent loop

Sem loop, é apenas uma chamada a LLM. Com loop + decisão dinâmica + autonomia, é um agente.

Cada iteração: o agente percebe o estado, raciocina sobre o próximo passo, age (chamada de tool ou resposta), e observa o resultado.

O loop é onde a inteligência emerge.

ReAct · o padrão default

Em vez de raciocinar tudo de uma vez, o agente intercala Thought → Action → Observation e decide o próximo passo a cada iteração. O trace cresce no contexto até o LLM emitir a resposta final.

Para single-agent, comece aqui. Yao et al. 2022 · arXiv:2210.03629

CASO DE USO · ReAct

Assistente de pesquisa científica

O agente recebe uma pergunta complexa, gera um thought (hipótese), executa uma busca (action), lê os resultados (observation) e decide se precisa de mais informação. Repete o ciclo com o trace acumulado no contexto até emitir a resposta fundamentada. Ideal para tarefas abertas onde o caminho não é conhecido antecipadamente.

Plan-and-Execute · separar planejar de fazer

Diferente do ReAct (decide passo a passo), Plan-and-Execute elabora um plano completo primeiro, depois executa.

VANTAGEM

Custo otimizado: LLM caro só planeja; executor barato roda os passos. Reduz custo total em tarefas longas.

DESVANTAGEM

Pouca adaptabilidade: se uma observação contradiz o plano, o agente segue cegamente até o fim.

CASO DE USO · PLAN-AND-EXECUTE

Due diligence financeira automatizada

O Planner (modelo grande) elabora o plano completo: 1. buscar dados, 2. analisar balanço, 3. comparar com o setor, 4. redigir conclusão. O Executor (modelo menor e mais barato) executa cada etapa mecanicamente. Ideal quando a tarefa é longa e bem estruturada — custo inferior ao ReAct puro e menos risco de desvio do objetivo.

Reflexion · auto-crítica verbal

Estende ReAct com auto-reflexão verbal após falhas. O agente escreve em memória episódica o que deu errado e por quê, e revisa nas próximas tentativas.

REFLEXION

Aprende em tempo de inferência. Não muda pesos. Funciona em qualquer LLM, só via prompt.

RLHF / FINE-TUNING

Aprende em tempo de treinamento. Muda pesos. Custo alto, específico ao modelo.

Shinn et al. (2023) · arXiv:2303.11366

CASO DE USO · REFLEXION

Agente de programação competitiva

O agente tenta resolver o problema, executa os testes e, ao falhar, escreve em memória episódica: "abordagem greedy falhou no caso de borda X por causa de Y". Na tentativa seguinte, lê a lição e muda a estratégia para programação dinâmica. Reduz erros repetidos sem retreinar o modelo — ideal quando tentativas iterativas são baratas e o critério de sucesso é objetivo.

Tree-of-Thoughts · busca em árvore deliberada

Em vez de seguir uma cadeia única, ToT expande múltiplas possibilidades, avalia cada uma com score, e prossegue pela mais promissora. Search clássico encontra LLMs.

Yao et al. (2023) · arXiv:2305.10601 · Cada thought é avaliado e expandido conforme score · Caminhos não-promissores são podados

CASO DE USO · TREE-OF-THOUGHTS

Resolução de problemas matemáticos complexos

Para o puzzle "Game of 24", o agente expande múltiplas equações candidatas, avalia cada uma com score de viabilidade, poda os caminhos piores e aprofunda os mais promissores. Use quando o espaço de soluções é grande, os passos intermediários são avaliáveis e exploração sistemática supera geração linear — o custo em tokens é alto, mas a qualidade da solução justifica.

Heurística de seleção do padrão

Comece com ReAct. Adicione Reflexion quando observar falhas repetitivas. Evolua para Plan-and-Execute quando o gargalo for o planejamento. Reserve ToT/LATS quando a qualidade da decisão justifica o custo.

1

ReAct

SEMPRE COMECE AQUI

Para a maioria dos casos com até 10-15 passos. Padrão estável, debuggable, suporte universal.

2

+ Reflexion

QUANDO HÁ FALHAS REPETIDAS

Adicione quando observar modos de falha recorrentes. Custo só em tempo de inferência, sem retraining.

3

Plan-and-Execute

QUANDO O PROBLEMA É O PLANO

Quando o planejamento é o gargalo e a execução é mecânica. LLM caro só no plano.

4

ToT / LATS

DECISÕES DE ALTO VALOR

Quando a qualidade computacional da decisão justifica o custo. Reserve para escolhas de alto impacto.

Cada combinação é um trade-off entre custo, latência, auditabilidade e flexibilidade.

Topologias · Padrões de coordenação · Comunicação · Decisão arquitetural

Single ou multi-agente?

A resposta default deveria ser sempre "um único agente". A evidência empírica é clara: multi-agente introduz custos que precisam ser justificados.

Comece com single-agent.

Migre para multi-agente apenas quando o single-agent provar-se insuficiente, e quando você puder articular qual benefício específico é o motor da decisão.

Benefícios que justificam a complexidade

Especialização

Subdomínios genuinamente distintos

Paralelismo

Subtarefas independentes

Robustez

Isolamento facilita recovery

Estrutura

Três topologias canônicas

Comece com chain. Promova a tree quando precisar de delegação hierárquica. Promova a graph apenas quando ciclos forem inerentes.

Chain

CADEIA LINEAR

Sequência fixa, dependências claras. Apropriado: extrair → classificar → publicar.

Tree

HIERARQUIA

Delegação multi-nível com escopo. Cada nó decide entre executar, delegar ou retornar.

Graph

REDE COM CICLOS

Conexões arbitrárias. LangGraph adota essa abstração. Flexibilidade máxima, custo: ciclos não-terminantes.

CASO DE USO · CHAIN

Sequência linear com dependências

Sistema de análise de documentos jurídicos: um agente extrai o texto do PDF, o seguinte classifica as cláusulas, o terceiro gera o resumo e o quarto formata o relatório final. Cada etapa só inicia após a anterior — rastreabilidade total, sem paralelismo desnecessário.

CASO DE USO · TREE

Hierarquia de delegação

Central de suporte técnico: agente root classifica o ticket (hardware, software ou rede) e delega ao sub-agente especialista. Se necessário, esse sub-agente pode sub-delegar para análise de logs. O resultado sobe pela árvore até chegar ao usuário com resposta consolidada.

CASO DE USO · GRAPH

Ciclos controlados com LangGraph

Gerador de pesquisa com revisão iterativa: agente busca informação → agente avalia relevância → se insuficiente, volta ao buscador; se suficiente, passa ao agente de síntese. Ciclos terminam ao atingir threshold de qualidade ou limite de iterações.

CASO DE USO · SUPERVISOR

Relatório de inteligência de mercado

O supervisor recebe a query do usuário, delega busca de dados ao agente web, análise ao agente analista e formatação ao agente editor. Consolida os três outputs em um relatório coerente — ideal quando os subtasks têm dependência do resultado final mas não entre si.

CASO DE USO · SWARM

Triagem hospitalar digital

Agente de triagem inicial coleta os sintomas e faz hand-off para agente de urgência ou clínica geral, que pode passar para agente especialista. Sem coordenador central — cada hand-off é baseado no estado clínico atual, permitindo adaptação dinâmica ao longo da jornada do paciente.

CASO DE USO · MERCADO

Processamento de jobs de ML em escala

Tarefas de treinamento são publicadas como lotes no marketplace. Agentes-GPU fazem bid baseado em capacidade disponível e custo atual. O job vai para o agente que oferecer melhor relação custo-velocidade — útil quando recursos são heterogêneos e variáveis.

CASO DE USO · DEMOCRÁTICO

Revisão de arquitetura de software

Três agentes revisores (segurança, performance, manutenibilidade) analisam o PR independentemente e votam. Aprovação requer maioria qualificada. Use quando a decisão é irreversível, tem alto impacto e o custo de LLM extra é menor do que o custo de um erro.

CASO DE USO · ROTEADOR

Assistente corporativo multi-domínio

A pergunta do usuário é embedada e comparada com os embeddings de cada domínio (RH, TI, Financeiro, Jurídico). O agente com maior similaridade vetorial recebe a query — sem LLM extra no roteamento, latência mínima e custo zero adicional.

CASO DE USO · BLACKBOARD

Detecção de intrusão em tempo real

Sensores de rede, logs de autenticação e alertas de firewall postam eventos no blackboard. Agente correlacionador detecta padrões de ataque cross-sensor (ex: falhas de login + scan de portas + exfiltração) e aciona resposta automática.

Supervisor pattern · em detalhe

Um orquestrador recebe a query, decompõe, delega a workers especializados e agrega o resultado.

Recovery centralizado. Auditoria fácil. Single point of failure mitigado por restart strategies.

Auditável Componível Recovery centralizado

CASO DE USO · SUPERVISOR PATTERN

Análise de contratos empresariais

O supervisor recebe o documento jurídico, delega leitura jurídica ao agente advogado, análise financeira ao agente CFO e verificação de compliance ao agente auditor. Consolida os três pareceres em um relatório de risco unificado — cada worker é especializado e independente, mas o supervisor garante que todos os ângulos são cobertos antes da entrega.

Comunicação e estado compartilhado

01

Message passing

Mensagens estruturadas direto entre agentes. Schemas tipados, não linguagem natural.

HTTP / gRPC / A2A protocol

02

Shared session state

Memória de sessão visível a todos os agentes do mesmo workflow. Reduz overhead de troca.

Redis / ETS / shared dict

03

Persistent memory

Memória de longo prazo curada. Promoção explícita de sessão para storage de longo prazo.

pgvector / Neo4j / MemGPT

79% das falhas multi-agent são de design, não técnicas. Schemas tipados e ownership claro de estado resolvem a maioria.

CASO DE USO · MESSAGE PASSING

Pipeline de NLP para e-commerce

Agente de intenção envia {intent:'compra', entities:{produto:'notebook', preco_max:5000}} ao agente de busca via gRPC. Schemas JSON tipados eliminam ambiguidade e permitem validação automática no gateway — use quando agentes são desenvolvidos por times diferentes e a interface precisa ser um contrato.

CASO DE USO · SHARED STATE

Booking de viagem multi-agente

Agentes de voo, hotel e carro lêem e escrevem no estado de sessão compartilhado {destino, datas, orçamento, reservas}. Cada agente completa sua parte sem precisar repassar o contexto inteiro — overhead zero de troca. Use quando os agentes operam no mesmo workflow e o volume de dados compartilhados é alto.

CASO DE USO · PERSISTENT MEMORY

Assistente de produtividade pessoal

Preferências, projetos em andamento e decisões passadas são promovidos de contexto temporário para pgvector no fim de cada sessão. Na conversa seguinte, o agente recupera o contexto relevante via busca semântica — sem precisar repassar todo o histórico no prompt.

Árvore de decisão arquitetural

Sete perguntas em ordem. Pare na primeira em que a resposta justifica avançar — não adicione complexidade sem motivo específico.

01

O fluxo é fixo e conhecido?

Sim → Workflow. Pare aqui.

02

Tarefa multi-passo com dependências previsíveis?

Sim → Plan-and-Execute. Plano explícito · custo controlado.

03

Single-agent é suficiente?

Sim → LlmAgent com ReAct. Default robusto.

04

Qualidade crítica · falhas recorrentes?

05

Sim → Multi-agent · Supervisor.

Sim → Swarm ou supervisor com paralelização.

07

CASO DE USO · PEV

Deploy de infraestrutura cloud

O Planner gera o plano Terraform, o Executor aplica as mudanças na AWS e o Verifier consulta a API para validar se o estado real corresponde ao desejado — acionando rollback automático se houver divergência. Use quando erros de execução têm custo alto e verificação independente é obrigatória.

CASO DE USO · ENSEMBLE

Diagnóstico médico assistido por IA

Três agentes especialistas analisam em paralelo: sintomas clínicos, resultados de exames e histórico familiar. Cada um produz um diagnóstico diferencial com nível de confiança, e o sistema agrega por votação ponderada. Use quando uma única perspectiva é insuficiente e o custo de tokens é justificável pelo risco da decisão.

CASO DE USO · SELF-IMPROVEMENT

Gerador de código com auto-teste

O agente gera a função → executa os testes unitários → analisa os erros → refina o código. Repete até todos os testes passarem ou atingir o limite de iterações. Reduz 80% dos bugs triviais antes de qualquer revisão humana — use quando o critério de qualidade é objetivamente mensurável.

CASO DE USO · BLACKBOARD

Monitoramento de segurança física

Câmeras IP, sensores de movimento e logs de acesso físico postam eventos no blackboard. O agente correlacionador detecta padrões suspeitos cross-sensor (acesso noturno + câmera offline + sensor ativo simultaneamente) e aciona alarme. Use quando múltiplas fontes de dados precisam ser correlacionadas em tempo real sem acoplamento direto.

CASO DE USO · MENTAL MODEL

Agente de trading algorítmico

Antes de executar uma ordem grande, o agente simula o impacto no livro de ordens via modelo de mercado interno. Se a simulação indica slippage acima de 0.3%, fragmenta a ordem ou cancela. A irreversibilidade da execução justifica o custo computacional da simulação — use em domínios onde ações não podem ser desfeitas.

CASO DE USO · META-CONTROLLER

Chatbot corporativo multi-especialidade

Classifica a pergunta por embedding e despacha para o agente Dev (código), RH (benefícios), Financeiro (reembolsos) ou Jurídico (contratos). Mantém histórico centralizado e escala para humano quando confiança < threshold. Use quando o volume de domínios é alto e treinar um agente generalista seria caro ou impreciso.

Alucinação em agentes

Definição

Quando agentes de IA geram informações falsas, inventam fatos ou tomam ações baseadas em dados inexistentes, acreditando que são verdadeiros.

Diferença crítica dos LLMs simples

LLMs Geram texto incorreto que humanos podem verificar — erro fica no plano textual.
Agentes EXECUTAM AÇÕES baseadas em alucinações — emails enviados, transações, arquivos deletados.
Impacto De erro textual para consequências reais — o custo deixa de ser cognitivo e se torna operacional e financeiro.

GuardRails

Camadas de validação e bloqueio que previnem comportamentos indesejados. Quatro tipos canônicos cobrem o ciclo completo.

INPUT

Entrada

Filtra prompts maliciosos, prompt injection e PII antes de chegar ao agente.

OUTPUT

Saída

Valida respostas: alucinação, toxicidade, dados sensíveis vazados.

BEHAVIORAL

Comportamental

Define ações permitidas: budget de tokens, limite de tool calls, escopo de operação.

TOPIC

Tópico

Mantém o agente dentro do domínio — rejeita off-topic e regiões proibidas.

Importância: compliance · segurança · confiabilidade

Trade-offs em segurança

Autonomia ⇄ Controle

Mais liberdade = mais risco. Governança restringe casos extremos mas reduz capacidade.

Performance ⇄ Segurança

GuardRails adicionam latência. Validações em cascata aumentam o custo total.

Generalidade ⇄ Especialização

Específico é mais seguro — agentes genéricos amplificam a superfície de ataque.

Custo ⇄ Proteção

Segurança tem preço: monitoramento, red-teaming, evals contínuas, infraestrutura redundante.

PARTE VI

Harness

Definição · Por que > modelo · Componentes · Memória · Edge · Produto · Takeaways

O que é um Harness?

Infraestrutura completa de software ao redor de um LLM — loop, memória, contexto, ferramentas, verificação e orquestração — que transforma a inteligência probabilística bruta em um sistema confiável capaz de executar tarefas complexas de horizonte longo.
O harness é o produto. O modelo é peça trocável.

Como o Harness opera · Claude Code como exemplo

● memória / loop  ·  ● ferramenta / contexto  ·  ● resposta final   |   Cada partícula representa um passo do ciclo agêntico

Por que harness > modelo

Modelos de fronteira chegam a cada 3–6 meses com ganhos marginais. Harness acumula know-how arquitetural intransferível que não viaja com novos pesos.

ERRO COMPOSTO

95%20 = 36%

Cadeia de 20 passos a 95% cada → 36% end-to-end. Harness reduz n ou insere verificação.

EVIDÊNCIA

Manus → ~US$ 2B

Meta pagou ~2B por um wrapper sobre Claude. LangChain +13,7 pts no Terminal Bench sem trocar modelo.

INEVITÁVEL

Alucinação

Xu et al. (2024): matematicamente impossível eliminar. Mitigação é arquitetural, não de prompt.

"Sem harness, um LLM responde uma pergunta. Com harness, ele termina um projeto."

Componentes do harness

O harness é isomórfico a um sistema operacional: LLM é a CPU, janela é a RAM, filesystem é o disco, ferramentas são syscalls.

COMPONENTE FUNÇÃO ANALOGIA SO
Agent Loop (ReAct) Pensamento → Ação → Observação Scheduler
Prompt Assembly Montar input ótimo por turno Memory manager (RAM)
Tool Registry Catálogo de ações + sandboxing Syscall interface
Memory Subsystem Working · Episódica · Semântica Filesystem + DB
Verification / Guardrails Schema validation, LLM-as-judge Type system
State Persistence Checkpoints, event sourcing Snapshot
Observability Tracing distribuído, custo, profiling OpenTelemetry

Memória do harness

Memória é cidadã de primeira classe. Quatro camadas inspiradas em hierarquia de cache, três tipos vindos da neurociência cognitiva.

CORE

RAM análoga

Objetivos, preferências, identidade. Sempre no contexto.

CONVERSATIONAL

Cache

Sumarização + busca semântica do histórico de sessão.

ARCHIVAL

Disco

BD vetorial. Ilimitada, recuperada sob demanda via retrieval.

EXTERNAL

Storage

Docs, PDFs, código. Acessíveis via tools específicas.

CoALA/Tulving: episódica (eventos) · semântica (fatos) · procedural (skills). + sleep-time compute: consolidação assíncrona entre interações.

Inferência na borda

Privacidade, latência, custo, disponibilidade. A trindade dos limites — memória, compute, energia — cede frente a PagedAttention, quantização Q4_K_M e speculative decoding.

KV-Cache · 5 eras

1. Contíguo (60–80% desperdício) → 2. PagedAttention/vLLM (96%+) ⭐ → 3. Distribuído → 4. LMCache hierárquico (3–10× latência) → 5. Offload NVMe (KVSwap).

Quantização & Métricas

Q4_K_M = melhor trade-off edge. Mistral 7B FP16→FP8: −8,5% latência, +33% throughput, −24% custo. Métricas edge: MBU (bandwidth utilization) e Joules/Token.

Modelo 7B em FP16 = 14GB. Flagship telefônico = 12GB total. Constraint primária: memória.

Harness como produto

Mesmos LLMs, harnesses diferentes → performances radicalmente diferentes. Mudar só o harness moveu 15 LLMs 20+ posições no TerminalBench 2.0.

01

LangChain Deep Agents

52,8% → 66,5% no Terminal Bench 2.0. Do Top 30 → Top 5. Loops de auto-verificação, doom-loop detection, context injection.

02

Manus (Meta dez/25)

5 rewrites em 6 meses. Descobertas: KV-cache hit rate primário, mascarar vs remover tools, filesystem como contexto, preservar erros no trace.

03

ICML 2026

Evoluir harnesses automaticamente via propositores ganhou +4,7 pts em raciocínio. Pesos congelados.

Padrões: context engineering · skills + progressive disclosure · subagentes (MapReduce) · verificação intermediária

Takeaways · Harness

01 · O valor mora no harness

Mesmo LLM + harness diferente = performance radicalmente diferente. Calibração é o moat.

02 · Erro composto é brutal

95%20 = 36%. Reduza n ou insira checkpoints. Verificação quebra a multiplicação exponencial.

03 · Pense como OS

LLM = CPU · janela = RAM · filesystem = disco · tools = syscalls. CoALA + BEAM/OTP mapeiam 1:1.

04 · Alucinação é arquitetural

Matematicamente inevitável. RAG, verificação, guardrails, human-in-the-loop — não prompt.

05 · Skills + progressive disclosure

Expor todas as tools destrói performance. Carregamento lazy mantém contexto enxuto e capacidade ilimitada.

06 · Tools são 70% do esforço

Não é prompt engineering que falha em produção — é tools mal desenhadas. Docstrings, idempotência, timeouts.

PARTE VII

Aplicabilidade

Onde brilham · Onde NÃO usar · Por setor · Maturidade · Defesa em profundidade

Quando agentes brilham

O retorno só compensa quando o problema tem características específicas — caso contrário, workflow ou single LLM resolve melhor.

01

Problema aberto e iterativo

Pesquisa, exploração, análise de documentos longos. O caminho até a resposta não é determinístico — o agente decide o próximo passo.

02

Decomposição valiosa

Tarefas divisíveis em subtarefas com ferramentas distintas. Coding, due diligence, recomendações personalizadas.

03

Tolerância a tentativa-erro

Domínios onde errar e corrigir é aceitável. Reflexion brilha aqui — custo de errar < custo de não tentar.

04

Personalização por contexto

Atendimento complexo, suporte técnico, triagem auxiliar. Cada query exige caminho diferente.

Quando não usar agentes

Nestes casos, a complexidade do agent loop não é apenas desnecessária — é prejudicial.

EVITE

FAQ e atendimento simples

Workflow ou RAG simples resolve. Agente adiciona latência e custo sem ganho real.

EVITE

Operações regulamentadas

Compliance, KYC, decisões médicas críticas. Auditabilidade de workflow > flexibilidade do agente.

EVITE

Latência rígida (< 2s)

Agentes têm cauda longa de latência. P99 frequentemente > 60 s. Use single LLM call.

EVITE

Volume sem variabilidade

Milhões de queries idênticas. Workflow + cache de prompts é ordens de magnitude mais barato.

"Prefira workflows quando puder, agentes apenas quando precisar." — Anthropic

Maturidade necessária · cinco dimensões

Antes de discutir framework ou modelo, avalie se a organização tem as condições para sustentar agentes em produção.

Estratégia

Caso de uso com ROI mensurável e clareza de sucesso.

Dados

Fontes acessíveis, limpas, com permissões claras.

Governança

Políticas de acesso, audit trails, accountability.

Observabilidade

Traces, métricas, alertas. Sem isso, debug é caos.

Aprendizado

Feedback loop e iteração contínua sobre falhas.

Falta em qualquer dimensão = projeto entra para os 80% que não chegam a produção.

Defesa em profundidade

Alucinação é matematicamente inevitável (Xu et al., 2024). A meta: limitar magnitude e garantir recuperabilidade.

CAMADA 1

Schema validation

JSON schemas tipados para inputs e outputs. Determinístico, barato.

CAMADA 2

Regras de negócio

Validações lógicas pré e pós-execução. Se invariantes violadas, aborte.

CAMADA 3

LLM crítico

Modelo separado avalia coerência. Detecta contradições semânticas.

CAMADA 4

Guardrails

Filtros temáticos, detecção de prompt injection, limites de escopo.

CAMADA 5

HITL crítico

Aprovação humana antes de operações destrutivas ou financeiras.

CAMADA 6

Reversibilidade

Tudo escrito, audit log, rollback. Recovery é arquitetural.

Camadas, não silver bullet · cada camada cobre um modo de falha distinto

Cinco padrões que funcionam

Anthropic · Building Effective Agents · os cinco workflows canônicos para produção
01

Prompt chaining

Output de uma chamada vira input da próxima. Determinístico, auditável. Use para extrair → resumir → publicar.

02

Routing

Classificador encaminha cada input ao agente especializado correto. Reduz custo, aumenta precisão.

03

Parallelization

Subtarefas independentes executadas em paralelo, agregadas no final. Ótimo para análises multi-fonte.

04

Orchestrator-workers

Supervisor decompõe dinamicamente, delega a workers especializados. O multi-agente real.

05

Evaluator-optimizer

Loop gerador-crítico até critério de qualidade ser atingido. Reflexion em produção.

Anti-patterns recorrentes

Os mesmos erros em organizações diferentes. Reconhecê-los cedo poupa meses de trabalho descartado.

EVITE

Multi-agente prematuro

Adicionar 5 agentes para um problema que single-agent + tools resolveria.

EVITE

Comunicação em prosa

Agentes trocando mensagens em linguagem natural não-tipada. Schemas JSON sempre.

EVITE

Sem timeouts duros

max_iters, max_time, cycle detection. Sem isso, loops infinitos e bills surpresa.

EVITE

Tools com escopo amplo

Uma única tool "execute_command" é pedido de catástrofe. Permissões mínimas.

EVITE

Memória global mutável

Estado compartilhado sem ownership claro. Race conditions e contexto envenenado.

EVITE

Avaliação ad-hoc

Sem suite de eval, sem regression testing. Drift do modelo passa despercebido.

Cinco coisas para levar

01

Agente exige autonomia + loop + decisão dinâmica. Sem os três, você tem workflow ou chamada LLM — ambos são apropriados em contextos diferentes.

02

Comece simples. Single-agent é o default. Multi-agente é otimização — exige justificativa específica de por que a complexidade vale o custo.

03

ReAct é o padrão default. Adicione Reflexion para falhas repetitivas, Plan-and-Execute para gargalo de planejamento, ToT/LATS para decisões de altíssimo valor.

04

Erro composto é matemática implacável. Investimento em harness, verificação e observabilidade compõe ao longo da execução — isso, não o modelo, é o que diferencia.

05

Aceitar que agentes vão errar. A meta: limitar magnitude e garantir recuperabilidade via defesa em profundidade — não eliminar erros.
